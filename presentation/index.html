Custom Instructions

Always-on context — global or scoped to specific files

📚 Global Instructions

copilot-instructions.md — injected into every chat request.

  • Tech stack, conventions, architecture decisions
  • Applied to chat and agent mode (not inline completions)
  • Keep it focused — too much noise reduces quality

🎯 File-Scoped via applyTo

Glob patterns match files relative to workspace root:

  • **/*.test.ts — all test files
  • src/api/** — entire API layer

Best used to differentiate frontend vs. backend rules, or apply language and framework-specific patterns.

---
applyTo: "src/components/**/*.tsx"
---

# React Component Guidelines

- Use functional components only
- Props interface named {Component}Props
- Wrap async data in Suspense

# Accessibility

- All interactive elements need aria labels
- Use semantic HTML elements

✅ Best Practices

  • Be specific: "use Vitest" not "use a testing framework"
  • Include reasoning: "Use date-fns — moment.js is deprecated"
  • Show code examples over abstract rules
  • Skip rules that linters/formatters already enforce

Prompt

One shot workflows invoked by the user via /slash-commands

---
mode: "agent"
tools: ["codebase", "github", "fetch"]
description: "Scaffold a new feature with tests"
---

# New Feature Scaffold

Break down the requested feature into:

1. **Data model** changes needed
2. **API endpoints** to create/modify
3. **UI components** to build
4. **Tests** to write

Follow the project conventions in our
copilot-instructions.md.

🚀 Meta Data

  • mode: ask, agent, or edit
  • tools: which tools the prompt can use
  • description: shown in the prompt picker
  • model: override the language model

💬 How to Invoke

Type / in chat to see available prompts, then add your own context after the command.

Custom Agents

Role-based personas

---
description: "Senior code reviewer"
tools: ["codebase", "githubRepo"]
handoffs:
  - label: "Update docs"
    agent: docs-writer
    prompt: "Document the reviewed code"
    send: false
---

# Code Reviewer Agent

You are a senior engineer doing code review.

## Review Checklist
- Type safety and error handling
- Security vulnerabilities (OWASP Top 10)
- Performance implications
- Test coverage adequacy

👥 Agent = Persona + Tools + Handoffs

  • Instructions define personality and expertise
  • Tools control what the agent can access
  • Handoffs create clickable buttons to transition
  • Can be invoked as subagents by other agents
  • File name becomes the agent identifier

🔄 How to Use

Select from the agents dropdown in Chat. The agent's instructions, tools, and handoffs apply to your conversation.

Agent Handoffs

Transitions between agents for guided workflows

---
description: "Generate an implementation plan"
tools: ["codebase", "fetch"]
handoffs:
  - label: "Start Implementation"
    agent: implementer
    prompt: "Implement the plan outlined above."
    send: false
  - label: "Request Review"
    agent: reviewer
    prompt: "Review the plan for feasibility."
    send: false
---
planner
→
Plan complete. Start Implementation Request Review
implementer
→
Implementation done! 🚀
reviewer
→
LGTM! Ready to ship! 🎉

Handoff fields

  • label — button text shown in chat
  • agent — target agent to switch to
  • prompt — pre-filled prompt for the next agent
  • send — auto-submit? (false = user confirms first)
  • model — optionally override the language model

How it works

After a chat response completes, handoff buttons appear as clickable suggestions. The user clicks to switch to the next agent with context and a pre-filled prompt. When send: true, the handoff is done automatically.

Subagents

Agents can orchestrate other agents for isolated, specialized work

Mode How Used Example
Top-level User switches to agent Plan agent, Migration agent
Subagent Orchestrator invokes Log analysis, code research, documentation

🔄 When Invoked as Subagent

  • Runs in its own context window
  • Receives an input prompt from the orchestrator
  • Returns just the response
  • Main agent's context stays clean

💡 Why Use Subagents?

Break complex tasks into focused subtasks. Each subagent works in isolation, keeping the orchestrator's context uncluttered and enabling parallel specialization.

Hooks

Automated actions triggered at key points in the agent lifecycle

{
  "github.copilot.chat.hooks": {
    "postProcess": [
      {
        "command": "npm run lint:fix",
        "description": "Auto-fix lint issues"
      },
      {
        "command": "npm run format",
        "description": "Format changed files"
      }
    ]
  }
}

💡 Use Cases

  • Auto-format code after every edit
  • Run linter and auto-fix issues
  • Execute tests on changed files
  • Validate builds haven’t broken

🔄 Lifecycle Events

  • preProcess — runs before agent processes a request
  • postProcess — runs after agent edits files
  • Commands run sequentially; failures surfaced in chat

Also compatible with Claude Code hooks!

MCP Servers

Connect Copilot to external tools and data via the Model Context Protocol

// .vscode/mcp.json
{
  "servers": {
    "github/github-mcp-server": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/",
      "gallery": "https://api.mcp.github.com/v0/...",
      "version": "0.13.0"
    },
    "microsoft/playwright-mcp": {
      "type": "stdio",
      "command": "npx",
      "args": ["@playwright/mcp@latest"],
      "gallery": "https://api.mcp.github.com/v0/...",
      "version": "0.0.1-seed"
    }
  }
}

🔌 What Is MCP?

An open protocol letting AI assistants call external tools, read resources, and receive prompts from any server.

🔎 Finding MCP Servers

Search @mcp in the VS Code Extensions tab to discover and install servers from the marketplace.

Putting It All Together

How the customization layers compose

Layer 1: Instructions (always active)
copilot-instructions.md + AGENTS.md + file-based *.instructions.md
Sets the baseline: tech stack, conventions, code style
↓
Layer 2: Prompts & Skills (on-demand)
*.prompt.md invoked via /slash-command • SKILL.md loaded when task matches
Repeatable workflows and portable capabilities
↓
Layer 3: Agents (role-based)
Select from agents dropdown — each with its own instructions, tools, and handoffs
Specialized teammates that orchestrate subagents
↓
Layer 4: Hooks (automated checks)
postProcess runs linting, formatting, tests after every agent edit
Quality gates that catch issues in real time
